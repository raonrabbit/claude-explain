# Explain 스킬 — 이해시키는 가이드북 만들기

주제(코드든 개념이든)를 받아, **사전지식이 전혀 없는 독자도 몰입해서 끝까지 이해할 수 있는** 가이드북 문서를 만든다.

## 북극성: "얼마나 잘 이해시키는가"

이 스킬의 유일한 성공 기준이다. 분량이나 형식이 아니라 **독자의 이해**가 목표다. 독자는 이 주제에 대해 아무것도 모른다고 가정한다. 그래서 두 가지가 항상 따라온다:

- **사전지식을 밖에서 찾게 하지 않는다.** 어떤 개념을 쓰기 전에 그 개념을 그 자리에서(또는 앞 챕터에서) 먼저 가르친다. "~는 아시죠?" 하고 넘어가지 않는다.
- **점층적으로 쌓는다.** 앞에서 가르친 것이 뒤를 이해하는 재료가 되도록, 쉬운 것에서 어려운 것으로 순서를 설계한다.

좋은 가이드북은 독자가 "왜 이걸 배우는지" 납득한 상태로 다음 장으로 넘어가게 한다. 지식을 나열하지 말고, 독자의 머릿속에서 그림이 그려지도록 안내하라.

## 실행 절차

### 1. 대상 파악 (여기서 대충 하면 나머지가 다 무너진다)

무엇을 설명하는지에 따라 준비가 다르다:

- **이 레포의 코드** — 관련 파일을 **실제로 읽어라**(Read/Grep/Glob). 어림짐작으로 쓰지 마라. 진입점부터 의존 관계를 따라가며 흐름을 파악한다. 설명에 인용할 실제 코드 조각과 그 위치(`파일:줄`)를 확보한다.
- **일반 개념/라이브러리/기술** — 확실히 아는 것만 쓴다. 최신 API나 버전별 동작처럼 불확실한 부분은 WebSearch/WebFetch로 확인한다. 추측을 사실처럼 적지 않는다.
- **둘의 혼합** — 대개 이 경우다. "이 코드가 무엇을 하는가"와 "이 코드를 이해하려면 무엇을 알아야 하는가"를 모두 다룬다.

이 단계에서 **독자의 출발점**도 정한다. 기본 가정은 "이 주제엔 문외한이지만 프로그래밍 자체는 아는 개발자"다. 사용자가 다른 수준을 지정하면 따른다.

### 2. 형식·구조 결정

- **형식**: 인자에 `--html`(또는 "html로" 같은 요청)이 있으면 HTML, 없으면 기본 **md**.
- **단일 vs 챕터 분할**: 주제의 범위와 복잡도를 보고 **직접 판단**한다. 한 흐름으로 읽힐 만큼 작으면 단일 파일, 사전지식 계단이 여러 개거나 독립적으로 소화할 덩어리가 여럿이면 챕터로 나눈다. 나눌 때는 각 챕터가 "한 번에 소화 가능한 한 입"이 되도록.
- 규모와 무관하게 **항상 폴더 하나로 감싼다** (아래 산출물 구조 참고).

### 3. `/explain-docs` 준비 & git local ignore

산출물은 레포 루트의 `/explain-docs/`에 만든다. 커밋에 잡히지 않도록 `.git/info/exclude`에 등록한다(개인 로컬 무시라 팀 `.gitignore`를 건드리지 않는다). 멱등하게:

```bash
# git 레포일 때만. /explain-docs/ 가 없으면 추가.
if [ -d .git ] && ! grep -qxF '/explain-docs/' .git/info/exclude 2>/dev/null; then
  printf '\n# explain 스킬 산출물 (로컬 전용)\n/explain-docs/\n' >> .git/info/exclude
fi
```

git 레포가 아니면 이 단계는 건너뛰고 `/explain-docs/`만 만든다.

### 4. 가이드북 작성

`/explain-docs/<주제-슬러그>/` 아래에 만든다. 슬러그는 주제를 알아볼 수 있는 짧은 kebab-case(예: `flightrail-globe-object`, `r3f-useframe`).

**산출물 구조:**

```
/explain-docs/<주제-슬러그>/
├── index.md            # 표지 + 목차 + "이 가이드북을 읽는 법" + 필요한 사전지식 지도
├── 01-<챕터>.md
├── 02-<챕터>.md
└── ...
```

단일 파일이어도 폴더로 감싸고 `index.md` 하나만 둔다. `index.md`에는 무엇을 다루는지, 어떤 순서로 읽는지, 다 읽고 나면 무엇을 이해하게 되는지를 담는다. 챕터가 여럿이면 각 챕터로 가는 링크 목차를 넣는다.

**각 챕터/설명의 리듬** — 고정 서식이 아니라 이 흐름을 지키라는 뜻이다:

1. **동기** — 이 장에서 무엇을, 왜 이해하게 되는가. 독자가 "그래서 이게 왜 필요한데?" 하지 않도록.
2. **먼저 알아야 할 것** — 본 설명에 필요한 사전 개념을 여기서 가르친다. 이미 앞 챕터에서 다뤘으면 한 줄로 상기시킨다.
3. **본 설명** — 개념은 비유·예시·(필요하면 ASCII/mermaid) 그림으로 구체화한다. 코드는 아래 철칙을 따른다.
4. **정리** — 한두 문장 요약, 또는 스스로 확인해볼 질문 하나.

**코드 설명 철칙:**

- 말로만 설명하지 말고 **실제 코드를 반드시 인용**한다. 레포 코드면 원문을, 그대로 붙이고 `파일:줄`을 밝힌다.
- 붙인 코드를 **읽는 데 필요한 문법·개념도 그 자리에서 가르친다.** (예: `useFrame(() => {...})`를 설명하려면 콜백이 매 프레임 호출된다는 것, 클로저가 무엇을 잡는지까지.)
- 긴 코드는 통째로 던지지 말고 의미 단위로 쪼개 한 조각씩 뜯어본다.

**출처·참고자료 표기 (신뢰성의 핵심):**

출처는 **두 층**으로 단다. 하단 종합만 있으면 독자가 "이 주장이 어느 문서 근거인지"를 스크롤해 찾아야 한다. 그래서 **주장이 나오는 그 자리에도** 링크를 건다.

- **① 인라인 인용 — 주장·개념·수치가 등장하는 바로 그 자리에 건다.** 두 방식 중 문맥에 맞게 택:
  - **개념어 자체를 파란 링크로**: `<a class="cite-link" href="URL" target="_blank" rel="noopener noreferrer">Route Handlers</a>` → 밑줄 + ↗로 "외부 문서로 넘어감"이 보인다. 개념 이름이 곧 링크일 때.
  - **문장 끝 ↗ 아이콘 마커**: `…5분 rolling window로 집계한다<a class="cite" href="URL" target="_blank" rel="noopener noreferrer" title="출처: 문서명">↗</a>` → 서술을 방해하지 않고 근거만 살짝 단다. 수치·동작을 진술한 문장 끝에.
- **② 하단 종합** — 각 챕터 끝 `## 참고 자료` 섹션에 그 장의 출처를 `source-tag`로 모아둔다(개요·재방문용). ①과 ②는 **둘 다** 둔다.
- **레포 코드 설명에서도** 적용한다. 코드 위치는 `파일:줄`로 짚되, 그 코드가 기대는 **일반 개념·프레임워크/라이브러리 동작**(예: "Next.js App Router가 ~", "React `useEffect` 클린업이 ~", "R3F `useFrame`이 매 프레임 ~")은 **공식문서를 인라인으로** 링크한다. 레포 내부 근거(코드)와 외부 근거(문서)를 구분해서 댄다.
- 외부 링크는 항상 **새 탭**: `target="_blank" rel="noopener noreferrer"`. 내부 챕터 링크는 같은 탭(일반 마크다운 링크).
- **불확실한 URL은 적지 않는다.** WebSearch로 정식 URL을 확인하고 태깅한다 — 죽은 링크는 신뢰를 깎는다.
- md 형식에선 인라인 인용을 일반 마크다운 링크 `[개념](url)`로 대체할 수 있으나, 새 탭·↗ 아이콘이 필요하면 위 HTML 앵커를 그대로 쓴다(md 뷰어도 링크는 정상 렌더).

### 5. 사용자에게 보고

만든 폴더 경로와 파일 목록, 어떤 순서로 읽으면 되는지 한 줄 안내를 전한다. HTML이면 브라우저로 열어보라고 안내한다.

## HTML 형식일 때

`assets/guidebook-template.html`을 사용한다. 이 템플릿은 자체 완결형 단일 파일로, 사이드바 목차·앵커 이동·코드 하이라이트·읽기 좋은 타이포를 제공한다(인터랙티브 기능은 넣지 않는다). 사용법:

1. 템플릿을 읽는다.
2. 본문을 **마크다운으로** 작성해 `<script id="content" type="text/markdown">` 안에 넣고, `<title>`과 표지 제목 placeholder를 채운다. 목차는 스크립트가 제목(h2/h3)에서 자동 생성하므로 따로 만들지 않는다.
   - ⚠️ **본문(코드 예제 포함)에 리터럴 `</script>` 문자열이 있으면** 브라우저가 콘텐츠 컨테이너를 거기서 닫아버려 그 아래가 통째로 렌더링 누락된다(대표 사례: XSS 예제). 반드시 `<\/script>`로 이스케이프한다(백슬래시는 코드 예제에 그대로 보이지만 표준 표기다).
3. 챕터가 여럿이면 각 챕터를 같은 템플릿으로 각각의 `NN-....html`로 만들고, `index.html`이 각 챕터로 가는 링크 허브가 되게 한다.
4. **챕터형 네비게이션·홈 링크·출처 태그는 템플릿에 내장돼 있다** (스타일·hover 포함). 각 챕터 HTML에서 클래스만 붙이면 된다:
   - 사이드바 상단에 `<a class="home-link" href="index.html">📚 전체 목차로</a>` — 상시 목차 링크. `index.html` 자신에는 넣지 않는다(템플릿의 그 줄을 지운다).
   - 본문 맨 끝(콘텐츠 `</script>` 직전)에 `<div class="chapter-nav">`로 **이전/목차/다음 3버튼**을 넣는다. 좌우 버튼엔 실제 이전·다음 챕터 제목을 적는다. 첫 장의 이전·마지막 장의 다음은 `<span class="disabled">`로 비활성 처리.
   - 참고 자료는 `<a class="source-tag" ... target="_blank" rel="noopener noreferrer">`로.

md와 HTML의 **내용·구조·철칙은 완전히 동일**하다. 형식만 다르다(단, 위 네비/홈/출처 태그는 HTML 전용 편의 요소다).

## 예시

**예시 1 — 레포 코드 설명**
입력: `/explain flightrail의 GlobeTileLayer가 지구본 타일을 어떻게 로딩하는지`
동작: `GlobeTileLayer.tsx`와 관련 파일을 읽고, 타일/슬리피 맵 좌표계 같은 사전지식을 먼저 가르친 뒤, 실제 코드를 인용하며 로딩 흐름을 단계별로 푼 md 가이드북을 `/explain-docs/flightrail-globe-tile-layer/`에 생성.

**예시 2 — 개념 설명, HTML**
입력: `/explain --html React Server Components가 뭔지 처음 보는 사람도 이해하게`
동작: 클라이언트/서버 렌더링 기본기부터 쌓아 올리는 챕터형 HTML 가이드북을 `/explain-docs/react-server-components/`에 생성.
